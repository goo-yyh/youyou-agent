# Codex Review Standard for YouYou Agent Design

## 目的

这份标准用于对 YouYou Agent 后续设计版本做重复、可比、可追踪的评审。核心目标不是写“主观观感”，而是用固定维度检查以下四件事：

- 需求有没有被完整承接
- 核心接口能不能真的实现
- 运行时状态机会不会漏边界
- 持久化、恢复、记忆、Hook、取消这些链路是否闭合

同一份标准应尽量复用到后续版本，避免每次 review 的口径漂移。

## 固定评审流程

### 1. 先读基线文档

- 必读需求文档，例如 `specs/0005_youyou_demand.md`
- 再读被评审设计文档，例如 `specs/0006_youyou_agent_design.md`
- 如设计文档引用其他文档，只加载与争议点直接相关的部分，不做无限扩散

### 2. 建立“需求 -> 设计”映射

至少核对以下块是否有明确设计落点：

- 注册接口：`ModelProvider`、`ToolHandler`、`Skill`、`Plugin`、`SessionStorage`、`MemoryStorage`
- 配置：`AgentConfig`、`SessionConfig`、默认值、消费方、校验规则
- 核心流程：`TurnLoop`、`ContextManager`、`PromptBuilder`、`ToolDispatcher`、`MemoryManager`
- 生命周期：build、new session、resume、send message、cancel、close、shutdown
- 事件：对外 `AgentEvent`、对内 Hook
- 错误：错误码、来源、可重试性、对外暴露方式
- 多模态、并发、取消、恢复、持久化、checkpoint

### 3. 先找高严重级别问题

优先检查下面这类问题：

- 设计在 Rust/异步语义下根本无法实现
- 违反单会话、顺序一致性、数据账本一致性
- 生命周期没有闭环，失败时会漏状态
- 文档 A 和文档 B 对同一件事给出冲突契约
- 核心可选组件的降级行为未定义，导致实现不可判定

### 4. 再评估中低严重级别问题

包括但不限于：

- 顺序、命名、错误码、标签、Schema 不一致
- 某些接口语义依赖隐式约定
- 测试和可观测性不足，后续难以验收
- 文档内部前后说法不一致

### 5. 最后再打分

- 先按 6 个维度分别打分，再汇总总分
- 不要先拍脑袋定总分
- 若仍存在高严重级别问题，总分通常不应高于 `84`

## 评分维度

总分 `100`，固定采用以下 6 个维度。

### 1. 需求覆盖度（30 分）

检查点：

- 是否完整覆盖需求文档中的必选模块和可选模块
- 是否对每个注册接口都给出了职责、约束、校验和运行时位置
- 是否覆盖配置项、默认值和消费方
- 是否覆盖多模态、事件、错误、并发、取消、恢复、存储、记忆
- 是否存在“需求未写，设计擅自扩展且未回写基线”的情况

打分锚点：

- `27-30`：需求映射完整，几乎无关键偏差
- `22-26`：大部分覆盖，存在 `1-2` 个中问题
- `15-21`：缺少多个模块，或存在 `1` 个高问题
- `0-14`：核心流程或核心接口缺失

### 2. 架构分层与模块边界（20 分）

检查点：

- 模块职责是否单一清晰
- 是否存在明显循环依赖或双向耦合
- 编排层、状态层、外部依赖层是否分离
- 公共类型、内部状态、trait port 的边界是否明确
- 是否为了“看起来完整”而引入不必要抽象

打分锚点：

- `18-20`：分层清楚，边界稳定，模块职责清晰
- `14-17`：整体可接受，但有若干边界含混
- `8-13`：模块职责交叉较多
- `0-7`：架构层级本身不稳定

### 3. 运行时正确性（25 分）

检查点：

- Session 生命周期是否闭环：new、resume、close、shutdown、失败回滚
- Turn 生命周期是否闭环：start、tool loop、cancel、complete、panic
- 单会话模型是否真的能被状态机保证，而不是只写在描述里
- Tool 执行顺序、并发策略、事件顺序、上下文顺序是否一致
- Context compact 的触发、重试、失败回退是否闭合
- 取消后哪些组件继续、哪些终止、哪些持久化，是否一致

打分锚点：

- `22-25`：关键路径闭环完整，没有明显竞态
- `16-21`：主体正确，但有中等边界遗漏
- `8-15`：存在高风险闭环缺失
- `0-7`：核心运行时模型不成立

### 4. 扩展性与接口契约（15 分）

检查点：

- trait 设计是否 object-safe、异步安全、可实现
- 所有权、生命周期、`Send + Sync`、`'static` 等 Rust 约束是否自洽
- Hook/Plugin/Storage 的扩展机制是否稳定
- 是否有“文档说可以这样用，但签名根本不支持”的情况
- 隐式约定是否被显式化

打分锚点：

- `13-15`：扩展接口稳定、约束清晰
- `10-12`：总体可用，但有若干契约歧义
- `5-9`：存在至少一个高风险接口缺陷
- `0-4`：核心扩展接口不可落地

### 5. 持久化、恢复与记忆闭环（15 分）

检查点：

- `SessionEvent` 是否足以支撑完整 replay
- 实时运行路径与恢复路径的“上下文账本”是否一致
- incomplete message、`ToolCall`、`ToolResult`、`SystemMessage` 是否可回放
- Memory 的加载、提取、更新、删除、checkpoint、幂等是否定义清楚
- 是否依赖未定义的存储行为

打分锚点：

- `13-15`：闭环清楚，可直接指导实现
- `10-12`：总体可行，但存在少量协议空白
- `5-9`：恢复或记忆有明显缺口
- `0-4`：无法可靠恢复或记忆协议不成立

### 6. 可测试性与可观测性（10 分）

检查点：

- 是否有统一错误码、来源、可重试性
- Event 是否足够支撑 UI、日志和调试
- 关键边界是否容易写成测试用例
- 文档是否指出关键验证场景
- 是否存在“只能靠实现猜”的行为

打分锚点：

- `9-10`：测试入口清晰，可观测性好
- `7-8`：大体可测，但仍缺少一些边界定义
- `4-6`：重要场景难测
- `0-3`：几乎不可验收

## 严重级别定义

### 高

满足任一条即可判为高：

- 核心接口在 Rust 中不可实现
- 会破坏单会话、上下文一致性、事件顺序、持久化账本
- 文档间契约冲突，导致实现基线不唯一
- 取消、恢复、关闭、panic 等核心生命周期缺少闭环

处理规则：

- 一般每个高问题会显著拉低总分
- 只要还有高问题未关闭，通常不建议进入实现
- 有高问题时，最终总分一般不应高于 `84`

### 中

典型情况：

- 语义未定义但可以补
- 顺序、标签、Schema、错误码不一致
- 某个子系统在不同章节写法不统一
- 存在实现分叉风险，但不至于立即阻塞编码

### 低

典型情况：

- 命名、措辞、样例、类型别名不统一
- 不影响核心设计成立，但会增加理解成本

## 固定检查清单

每次复评都应按下面清单至少扫一遍。

### A. 注册与配置

- 至少一个 `ModelProvider` 是否可编译期或运行时保证
- `provider id`、`model id`、`tool name`、`skill name`、`plugin id` 是否都有唯一性校验
- `default_model`、`compact_model`、`memory_model` 的校验和错误码是否清楚
- 可选组件未注册时的降级行为是否明确

### B. Session 生命周期

- `new_session()` 是否先校验、再 claim、再回滚，顺序是否合理
- `resume_session()` 是否对称处理互斥、失败、恢复状态
- `close()` 和 `shutdown()` 是否共用一套内部关闭流程
- 关闭后对象是否失效，是否存在旧句柄继续可用的问题
- 失败路径是否会留下脏状态

### C. Turn 生命周期

- `send_message()` 是否防并发
- `run_turn()` 的输入参数是否覆盖 cancel、event、state、依赖注册表
- panic 后是否能恢复 `turn_in_progress`
- cancel 后最后一个事件是否稳定
- 关闭令牌链路是否唯一、可追踪、可验证

### D. Tool 与 Hook

- Tool 批次并发/串行策略是否与需求一致
- `ToolCallStart` / `ToolCallEnd` / 持久化 / 上下文顺序是否一致
- Hook 的 `Continue` / `ContinueWith` / `Abort` 是否真有可实现语义
- Hook 对 payload 的修改是否强类型、可验证、不可静默失效
- 未知 Tool、Tool timeout、Hook abort 的策略是否落到主流程

### E. Context 与 Prompt

- Prompt 拼装顺序是否与需求一致
- 新增段落或 XML 标签是否写回基线
- 压缩触发、压缩失败回退、重试次数是否定义清楚
- 实时路径与恢复路径的 `Message` 序列是否完全一致
- 所有进入上下文的 `SystemMessage` 是否都能持久化或确定性重建

### F. Storage 与 Memory

- `SessionEvent` 是否足以完整回放
- incomplete assistant message 如何处理
- `SystemMessage` 的落盘和重建规则是否清楚
- Memory 的 `search / upsert / delete / list` 语义是否完整
- bootstrap memory、relevant memory、checkpoint memory 是否区分清楚
- checkpoint 计数器的语义是否唯一

### G. 错误与事件

- 每个主要错误是否有 `code / message / retryable / source`
- build 阶段与运行阶段错误边界是否清楚
- `STORAGE_ERROR`、`TOOL_NOT_FOUND` 等错误是否真的进入主流程，而不只是枚举定义
- Event 是否足够支撑 UI 和排障
- 顺序保证是否落到设计，而不是口头描述

## 分数解释

- `90-100`：设计稳定，可进入实现。只允许存在低问题
- `85-89`：基本可实现，存在少量中问题，但无高问题
- `70-84`：基础设计成立，但还不适合直接开工，需要先修正高问题或多处中问题
- `0-69`：核心契约不成立，建议先重构设计

## 复评通过条件

后续版本想判定为“通过”，建议至少满足：

- 总分 `>= 85`
- 不存在高严重级别问题
- 需求文档与设计文档对核心协议没有冲突
- Session、Turn、Tool、Memory 这四条主链路都能画出闭环
- 关键边界已经能直接转化为测试用例

## 复评执行方式

后续修改 `0006` 后，建议固定按下面顺序复评：

1. 先按上面的 6 个维度重新打分。
2. 再逐项核对 `R1-R7` 是否已关闭。
3. 若任一 `R1-R4` 仍未关闭，结论默认不能写“通过”。
4. 若 `R1-R7` 全部关闭，再看是否仍有新的同级别问题出现。
5. 复评结论中必须显式写出：
   - 哪些旧问题已关闭
   - 哪些旧问题仍然打开
   - 是否新增了新的高/中问题
